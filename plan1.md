# 编译原理大作业项目小组计划：C-- 编译器前端实现

## 一、项目概览
- **截止时间**：2026 年 05 月 20 日 11:59PM（GMT+8）。
- **目标**：实现 C-- 语言（C 语言子集）的编译器前端，输出标准 LLVM IR（`.ll`）文件。
- **核心技术栈**：C++（推荐）、自动机理论、SLR 语法分析、LLVM 架构。
- **关键约束**：严禁使用 Lex/Yacc/ANTLR 等自动生成工具；必须手动实现 DFA 确定化与最小化。

## 二、任务目标与范围
1. **词法分析器**：基于自动机理论实现 DFA 确定化与最小化，输出单词符号序列并维护符号表。
2. **语法分析器**：采用 SLR 方法，构造 LR(0) 项目集规范族、FIRST/FOLLOW、SLR 表，输出规约序列并构建 AST。
3. **中间代码生成**：补全 visitor 函数，遍历 AST 并调用中端接口生成 LLVM IR（`.ll`）。

## 五、接口统一与协作规范（统一形式）
### 1. 目录结构统一
- `src/lexer`：词法分析器与 DFA 相关代码。
- `src/parser`：SLR 表构建、分析驱动与 AST 生成。
- `src/ir`：visitor 与中端对接代码。
- `include/common`：Token、AST、符号表等公共头文件。
- `tests`：词法/语法/中端测试用例与期望输出。

### 2. Token 与符号表统一结构
- `Token` 统一字段：`type`、`lexeme`、`line`、`column`。
- `TokenType` 枚举：`KW`、`OP`、`SE`、`IDN`、`INT`、`FLOAT`、`EOF`。
- 符号表条目统一字段：`name`、`kind`、`type`、`scope`、`value`（可选）。

### 3. AST 结点统一形式
- 每个 AST 结点包含：`nodeType`、`children`、`token`（可选）、`line`、`column`。
- 规约序列中产生式编号与附录文法严格一致。

### 4. 模块接口统一（输入/输出约定）
- **Lexer → Parser**：提供 `vector<Token>`，末尾追加 `EOF`。
- **Parser → IR**：提供 `AST` 根结点与符号表（全局+局部）。
- **IR 输出**：仅输出 `.ll` 文件，不输出 `.bc`。

### 5. 输出格式统一
- **词法输出**：`[单词符号] [TAB] <[种别],[内容]>`，关键字不区分大小写。
- **语法输出**：`[序号] [TAB] [栈顶符号]#[面临输入符号] [TAB] [执行动作]`。
- **动作枚举**：`move` / `reduction` / `accept` / `error`。

## 六、核心开发规范
### 1. 词法规范
- 关键字（`int`, `void`, `return`, `const`, `main`, `float`, `if`, `else`）不区分大小写。
- 运算符（`+ - * / % = > < == <= >= != && ||`）与界符（`( ) { } ; ,`）需完整识别。
- 标识符、整数、浮点数匹配规则严格遵循附录。

### 2. 语法规范
- 必须覆盖附录中 36 条文法规则，保持规则编号一致。
- SLR 分析需输出规约过程与语法错误信息。

### 3. 中端衔接
- visitor 以 `visitChildren` 递归访问 AST。
- 使用符号表结构 `s.variable()` 做重复命名检查。
- 调用 `GlobalVariable::create` 等中端接口完成 IR 构造。
